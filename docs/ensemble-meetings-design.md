# Ensemble Meetings Design

This note captures the next worthwhile meetings-view step for Yggsii so the overnight pause does not turn into context loss.

## Problem

The current meetings view works well for simple two-character scenes and is decent at pointing back to the active scene when a larger cast is present.

Where it still feels thin is ensemble handling. Pair cards are useful, but they flatten a larger scene into fragments. That makes it harder to answer questions like:

- who is in the room right now
- which relationships are central versus incidental
- whether the active scene is a tight triangle, a split cluster, or a full-cast collision
- which pair card should matter most when a scene includes many characters

## Design goal

Keep meetings view lightweight, but let it describe ensemble scenes as small social structures rather than a bag of isolated pairs.

## Proposed addition

Add an active scene ensemble summary panel above the pair list whenever the current scene has three or more characters.

The panel should show:

- scene title
- optional time label
- optional location
- ordered cast list
- count of distinct pairs generated by the cast
- quick jump back to scene editing

It should also include a tiny structural summary:

- "Core cast" for characters explicitly marked present in the current scene
- "Pair coverage" showing how many pair cards below are generated by this scene
- "Most connected characters" based on how many pairings each cast member produces inside the active scene

## Why this is the right next slice

It improves comprehension without inventing a whole graph system.

It builds directly on the current meetings model.

It gives larger scenes their own readable identity instead of forcing users to mentally reconstruct them from pair rows.

It creates a cleaner bridge toward future relationship modelling if that ever becomes first-class.

## Suggested implementation shape

1. Add a helper that derives active-scene ensemble metadata from the selected scene.
2. Render a dedicated ensemble summary card in meetings view when cast size is at least three.
3. Reuse existing jump-to-scene behaviour rather than creating a new navigation path.
4. Keep pair cards below, but visually mark which rows are generated by the active ensemble.
5. Avoid persistence changes. This should be a pure derived-view improvement.

## Guardrails

- Do not add graph libraries.
- Do not introduce explicit relationship editing yet.
- Do not duplicate the full scene editor.
- Keep the panel readable in both light and dark themes.

## Acceptance check

A user opening meetings view from a five-character scene should be able to understand the active ensemble at a glance before reading any pair card, then jump back to scene editing if needed.

## Follow-on ideas, not part of this slice

- rank pair cards so active-scene pairs appear first
- cluster pair cards by active scene versus historical overlap
- add relationship labels once explicit relationship records exist
- surface "solo bridge" characters who connect otherwise separate subgroups
